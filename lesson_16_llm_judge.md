# Lesson 16: LLM-as-a-Judge

In Lesson 15, we discovered that traditional evaluation metrics (like Exact String Match or ROUGE) fail for Generative AI because there are a hundred correct ways to say "The store opens at 9 AM." We need a smarter judge.

## 1. Business Context

**Who requested this?**
The QA Team and Data Scientists.

**Why?**
Human evaluation is too slow. Traditional metrics are too inaccurate. We need an automated system that can understand the *meaning* of the agent's answer and compare it to the ground truth.

**Business Impact**
Enables 100% automated, high-fidelity testing of the AI system, unlocking true CI/CD.

**Customer Problem**
"My agent failed the benchmark test, but when I read the answers, they were actually correct. The test is broken."

**ROI & Metrics**
*   **Evaluation Accuracy:** Achieve > 90% correlation between the automated Judge's score and human expert scores.

---

## 2. Simple Analogy

Imagine a high school essay grading system.
*   **Traditional Metric (ROUGE):** A computer program counts how many words in the student's essay match the teacher's answer key. (Student fails because they used synonyms).
*   **LLM-as-a-Judge:** You hire an English Professor (a larger, smarter LLM like GPT-4 or Llama-3-70B). The professor reads the student's essay, reads the answer key, and uses their intelligence to say, "Yes, the student understood the concept. Grade: A."

---

## 3. First Principles

*   **What:** Using a powerful LLM to grade the output of your application LLM.
*   **Why:** To semantically evaluate unstructured text outputs against multiple criteria (Correctness, Relevance, Toxicity).
*   **How:** By writing a highly specific "Judge Prompt" and passing the `question`, `agent_answer`, and `ground_truth` into it.
*   **When:** During automated evaluation runs.
*   **Tradeoffs:** Cost. Using an LLM to judge every answer effectively doubles your inference cost during testing. Also, the Judge can hallucinate and give a wrong score.
*   **Failure Scenarios:** "Judge Bias." LLMs tend to give higher scores to answers that are longer, even if they are wrong. They also tend to favor answers generated by themselves (e.g., GPT-4 giving higher scores to GPT-4 outputs).

---

## 4. Internal Working

1.  **Test Run:** Agent outputs: "We open at 0900 hours." Ground Truth: "Store hours are 9 AM."
2.  **Judge Prompting:** The evaluator script calls the Judge LLM with this prompt:
    ```
    You are an expert evaluator. Compare the Student Answer to the Ground Truth.
    Is the semantic meaning identical? Output YES or NO.
    Ground Truth: Store hours are 9 AM.
    Student Answer: We open at 0900 hours.
    ```
3.  **Judge Output:** "YES"
4.  **Metric Logging:** MLflow logs a 1.0 (Correctness Score) for that row.

---

## 5. Databricks Implementation

Databricks **Mosaic AI Model Evaluation** provides built-in LLM-as-a-Judge metrics. You simply configure it by providing the endpoint of your Judge model (usually the most capable model you have, like Databricks DBRX or Llama 3 70B). Databricks handles the complex prompt engineering behind the scenes for metrics like `answer_correctness` and `retrieval_relevance`.

## 7. Explain Every Line of Code

Looking at `src/evaluation/judge.py`:
*   `from mlflow.metrics.genai import answer_correctness, relevance`: We import the specific GenAI metrics from MLflow.
*   `judge_endpoint`: We define which model acts as the judge. It should ideally be a larger, more capable model than the one used in the application.
*   `correctness_metric = answer_correctness(...)`: We instantiate the metric. Databricks will use this metric to prompt the Judge endpoint behind the scenes.
*   `extra_metrics=[correctness_metric]`: We pass this custom metric into `mlflow.evaluate`. Now, alongside the basic string metrics, MLflow will compute the LLM-judged score.

## 9. Production Problems

**The Problem: Evaluating Retrieval (Not just Generation)**
The agent answers incorrectly. Was it the LLM's fault (hallucination), or the Retriever's fault (brought back the wrong PDF)?
*   **The Senior Solution:** You must evaluate the pipeline in parts. Use LLM-as-a-Judge for two distinct metrics:
    1.  `retrieval_relevance`: Does the retrieved chunk actually contain the answer to the question?
    2.  `answer_faithfulness`: Is the agent's final answer strictly derived from the retrieved chunk?
    By separating these, you can pinpoint exactly which part of the RAG architecture failed.

---

## 10. Design Decisions

**Custom Prompts vs Built-in Metrics**
Databricks provides built-in judge prompts. However, for specialized industries (like Healthcare), the built-in prompts might not be strict enough. You can pass a custom `prompt_template` to the `answer_correctness` metric. 
*Decision:* Start with the built-in metrics to establish a baseline. Only write custom judge prompts if the Judge's scores are consistently disagreeing with human QA reviewers.

---

## 11. Cost Engineering

*   **Judge Costs:** If your eval dataset is 10,000 rows, evaluating `correctness`, `relevance`, and `toxicity` means 30,000 calls to the Judge LLM. If the Judge is an expensive model, this could cost hundreds of dollars per CI/CD run.
*   **Optimization:** Use smaller, specialized models as judges. Research shows that fine-tuned 8B parameter models can often judge specific tasks just as well as 70B models, at 1/10th the cost.

## 12. Interview Preparation (Senior Level)

1.  **Architecture:** "How do you evaluate if a RAG agent's retriever is failing vs if the generation step is failing?" (Answer: Evaluate relevance and faithfulness independently).
2.  **System Design:** "Design a cost-effective evaluation pipeline for a system receiving 1 million queries a day." (Answer: Randomly sample 1% of queries, log them to a Delta table, and run an offline batch LLM-as-a-Judge process nightly).
3.  **Tradeoffs:** "What are the risks of using LLM-as-a-Judge, and how do you mitigate them?" (Answer: Judge bias and hallucination. Mitigate by doing periodic human-in-the-loop audits of the Judge's decisions).
4.  **Coding:** "Explain how to configure a custom LLM-as-a-Judge metric using MLflow."
